01 — Problem Statement

The Broken Reality of Traditional Hiring

Every day, HR teams drown in hundreds of resumes for a single opening. Manual screening is slow, biased, expensive — and existing tools make things worse.

"A typical recruiter spends only 6–7 seconds per resume. With 200+ applicants per role, most qualified candidates are rejected before anyone truly reads their profile."
⏰

Time-Consuming Manual Screening

Reading 100–500 resumes per job takes 10–40 hours. Recruiters burn out and qualified candidates get missed purely due to volume.

🎯

Inconsistent & Biased Evaluation

Human reviewers apply different standards, get fatigued, and unconsciously favor certain formats — creating unfair, unreliable shortlists.

🔒

Privacy & Data Security Risks

Sending sensitive candidate PII to third-party cloud AI APIs risks data leakage, GDPR violations, and loss of candidate trust.

💸

High Cost of Existing AI Tools

Per-resume API billing. Screening 500 resumes against 3 JDs = 1500 API calls. Costs spiral quickly at scale.

📄

ATS Keyword Stuffing

Traditional ATS systems only match exact keywords. Candidates game the system — shortlists become full of false positives.

🤷

No Actionable Feedback

Recruiters get Yes/No decisions with zero transparency. No skill gap analysis, no score breakdown, no understanding of why.

02 — Our Solution

How We Solve Every Problem

The Resume Ranker solves all of these issues through a Hybrid Two-Phase Architecture with multi-model AI orchestration — instant, fair, transparent, and private at scale.

⚡

Instant Screening at Scale

Phase 1 processes resumes in parallel batches of 4 using deterministic algorithms. Results appear in 2–4 seconds per resume. No AI calls needed for ranking.

🧠

Semantic Understanding

Transformer-based embeddings understand meaning, not just words. "Distributed systems design" matches "scalable backend architecture" even with zero keyword overlap.

🔐

🔄

📊

✉️

Automated Bulk Email System

Phase 1 — Detailed Step Breakdown

Every resume goes through these 6 steps server-side before any AI call.

1

Document Parsing

File (PDF or DOCX) received as Base64 Data URI. Routes to pdf-parse (PDFs) or mammoth (Word). Raw text extracted, cleaned, normalized. Regex extracts candidate name and email. No email = candidate skipped with warning.

lib/processing/document-parser.ts
2

Semantic Embeddings

JD and resume text converted to dense numeric vectors. Primary: Gemini gemini-embedding-2 (768-dim). Fallback: Local ONNX all-MiniLM-L6-v2 (384-dim). Cosine similarity → Semantic Score 0–100. In-memory LRU cache (200 entries) prevents re-computation.

ai/embeddings/embedding-service.ts
3

Skill Matching

Regex-based extraction against skills taxonomy from both JD and resume. Handles exact matches, partials ("React" = "ReactJS"), and synonyms ("ML" = "Machine Learning"). Returns matchedSkills[] and missingSkills[]. Score = matched/required ratio × 100.

lib/skills/skill-matcher.ts
4

ATS Analysis

Rule-based ATS checker: presence of standard sections (Experience, Education, Skills, Contact), date formatting, length check, absence of tables/graphics that confuse ATS parsers, keyword density, machine-readability. Returns 0–100 score + issue list.

lib/ats/ats-analyzer.ts
5

Experience Heuristic

Date range detection in work history (e.g., "Jan 2018–Mar 2022" = 4.2 years). Total estimated years compared to JD requirements. Configurable baseline (default 70/100). Designed for easy enhancement with deeper NLP date parsing.

rank-candidates-progressive.ts:85
6

Composite Score & Ranking

Weighted formula: Semantic(45%) + Skill(25%) + Experience(15%) + ATS(10%) + LLM(5%). Clamped to 0–100 → letter grade. All candidates sorted descending by score. Quick non-AI feedback summary generated immediately from matched/missing skills.

lib/scoring/composite-scorer.ts

Two Orchestration Modes

🔄 Auto-Fallback Mode

The system tries all configured models in priority order. If Gemini is rate-limited or down, it instantly falls back to GPT → Claude → Grok → Groq → Local. Recruiter sees a result regardless — the system never says "AI unavailable."

// Auto mode: tries ALL enabled models in cascade
if (isEnabled('gemini')) modelsToTry.push(gemini);
if (isEnabled('gpt')) modelsToTry.push(gpt);
if (isEnabled('claude')) modelsToTry.push(claude);
// ... continues until one succeeds

🎯 Manual Mode

Recruiter explicitly selects their preferred model from the Profile → API Configuration page. System uses only that model — no automatic failover. Useful when you want to control exactly which AI analyzes your resumes.

// Manual mode: only the chosen provider
const provider = config.activeModel; // e.g. 'gemini'
modelsToTry.push({ provider, model, key });
// No fallback — manual choice respected

Robust JSON Repair Pipeline

Different AI models return JSON in different formats. The orchestrator runs a 4-stage repair pipeline to guarantee valid structured data from any model.

🧹

Stage 1: JSON Cleaning

Strips markdown code blocks (```json ... ```), extracts JSON from prose, escapes unescaped control characters (newlines, tabs) inside string values.

🔑

Stage 2: Key-Based Extraction

If standard parse fails, uses known schema keys (strengths, improvements, resumeAdditions) as anchors to locate and extract values directly from raw text — even if surrounding JSON is malformed.

🔧

Stage 3: Truncation Repair

Detects unclosed strings, brackets, and braces. Appends missing closing characters to repair truncated responses caused by token limit cutoffs.

🗺️

Stage 4: Key Normalization

Maps 40+ synonym key variants across models: weaknesses → improvements, bulletPoints → resumeAdditions, candidateStrengths → strengths. Works with all 6 models.

📦

Stage 5: Nested Unwrapping

Some models wrap their response in an extra object layer. The orchestrator detects single-key wrappers and automatically unwraps the inner data before validation.

✅

Stage 6: Zod Validation

Final step: Zod schema validates the cleaned, repaired, normalized JSON. On schema mismatch, falls back to raw parsed object. On complete failure, tries the next model in cascade.

07 — Local LLM & Privacy Mode

Running AI Locally with Ollama

For maximum data privacy, recruiters can run the entire AI pipeline locally using Ollama — an open-source tool that runs LLMs directly on your machine. No internet required for AI inference. Zero data sent anywhere.

Why Qwen 2.5 7B Instruct? It's specifically trained to follow structured instructions and return valid JSON — exactly what resume screening requires. At 7 billion parameters it's powerful enough for nuanced feedback yet lightweight enough to run on a standard laptop or workstation GPU without specialized hardware.
🎯

Instruction-Tuned by Design

Qwen 2.5 7B Instruct is fine-tuned specifically to follow precise formatting instructions. When told "return only valid JSON with keys strengths, improvements, resumeAdditions" — it reliably complies, unlike base models that often hallucinate structure.

🏎️

Optimal Size / Performance Ratio

7B parameters = fast inference on consumer hardware (4–8GB VRAM). Generates typical feedback in 8–15 seconds on a mid-range GPU. Smaller (3B) models sacrifice quality; larger (13B+) require enterprise GPUs. 7B is the sweet spot.

🔏

Complete Data Privacy

When using Ollama, resume text is processed entirely on your local machine. The Ollama HTTP server runs at localhost:11434. The model-router calls it directly via LAN — no resume data ever traverses the internet.

💰

Zero Ongoing Cost

Download once (~4.1GB for qwen2.5:7b-instruct), run forever. No per-call billing, no subscription fees, no quota limits. Process 10,000 resumes for free after the initial download.

🌐

Works Offline

Once Ollama is running and the model is downloaded, Phase 2 AI feedback works with no internet connection. Critical for organizations with strict network isolation policies or air-gapped environments.

🔧

Fully Configurable

The model name, Ollama server URL, and temperature are all configurable via the Profile → API Configuration page. Swap from qwen2.5 to llama3.2 or mistral with a single field change.

// Step 1: Generate 768-dimensional vectors (Gemini Primary)
JD_vector = [0.23, -0.41, 0.87, ... ] // 768 numbers
Resume_vector = [0.19, -0.38, 0.82, ... ] // 768 numbers

// Step 2: Cosine Similarity (angle between vectors)
cosine(A, B) = dot(A,B) / (|A| × |B|) // Range: -1 to +1

// Step 3: Sigmoid-Tanh Normalization to 0–100
linear = ((cosine + 1) / 2) × 100
adjusted = 50 + 50 × tanh((cosine - 0.4) × 2) // High contrast for comparison
finalScore = (linear + adjusted) / 2 // 0–100 score

Why sigmoid-tanh normalization? A raw cosine similarity of 0.85 and 0.90 would compress to very similar scores on a linear scale. The tanh mapping creates high contrast in the 0.3–0.8 range where most real candidate matches fall — making it easier to differentiate a "Good" from an "Excellent" match in the ranking table.

Dual Embedding Strategy: Cloud + Local

🌐 Primary: Gemini Embedding API (768-dim)

State-of-the-art cloud embeddings from Google. 768 dimensions give twice the representational capacity vs local model. Used automatically when GOOGLE_API_KEY is set. Batches of 10 texts processed together for efficiency.

768 DimensionsCloud AccuracyREST APIAuto-cached

🖥️ Fallback: all-MiniLM-L6-v2 ONNX (384-dim)

Downloaded once (~23MB from HuggingFace) as an ONNX model. Runs entirely in Node.js via @xenova/transformers. Mean-pooling + normalization applied. Zero cloud dependency. Works offline. Loads in ~2s once, then cached in memory for the session.

384 DimensionsRuns in Node.jsONNX RuntimeOffline-capable

📦 In-Memory LRU Cache (200 entries)

Every generated embedding is cached by the first 100 characters of its text. When processing 100 resumes against the same JD, the JD is embedded once and cached — saving 99 redundant API calls. Cache is a JavaScript Map with FIFO eviction when it hits 200 entries. Cache hit returns in 0ms vs 200–800ms for a fresh Gemini API call.

12 — Privacy & Security

Privacy-First vs Traditional Tools

Traditional AI recruitment tools send every resume to third-party servers. Our architecture keeps sensitive data local wherever possible and gives recruiters full control.

✕ Traditional Cloud-Only AI Recruiters

  • Every resume uploaded to vendor's cloud servers
  • Processed on OpenAI / Azure / unknown infrastructure
  • Unknown data retention and model training policies
  • High GDPR compliance risk with candidate PII
  • Per-call billing — expensive at 500+ resumes
  • Single point of failure — API down = zero screening
  • No control over model changes or output format
  • Vendor can theoretically access all candidate data

✓ Our AI Recruiter Approach

  • Phase 1 runs 100% on your own Next.js server
  • Local Ollama mode: zero data sent anywhere
  • Resume data scoped to authenticated user in Firestore
  • GDPR-friendly: no third-party processing by default
  • Phase 2 sends only truncated text (max 3000 chars)
  • Firestore security rules: per-user strict isolation
  • ONNX fallback: works offline if all cloud APIs fail
  • Recruiter explicitly controls when & which AI runs
